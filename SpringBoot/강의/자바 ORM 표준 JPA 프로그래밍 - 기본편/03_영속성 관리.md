# 영속성 관리

## 영속성 컨텍스트

영속성 컨텍스트 : 엔티티를 영구 저장하는 환경이라는 뜻으로 논리적인 개념

```EntityManager.persist(entity)``` : 엔티티를 DB가 아니라 영속성 컨텍스트에 저장하는 것

<br>

### 엔티티의 생명 주기

+ 비영속(new/transient) : 영속성 컨텍스트와 전혀 관계가 없는 새롭게 생성한 상태 (entityManager에 넣지 않고 객체를 생성만 한 상태)
+ 영속(managed) : 영속성 컨텍스트에 의해 관리되는 상태
+ 준영속(detached) : 영속성 컨텍스트에 저장되었다가 분리된 상태
+ 삭제(removed) : 삭제된 상태

<br>

### 영속성 컨텍스트의 이점

+ 1차 캐시 : 영속성 컨텍스트 내부에는 1차 캐시가 있어 조회 시 DB보다 1차 캐시를 먼저 확인함 하지만 하나의 트랜잭션에서만 사용되는 단위이기 때문에 엄청난 성능의 이점을 얻을 수는 없음
+ 영속 엔티티의 동일성(identity) 보장 : 같은 데이터를 조회한 후 == 비교 시 같다고 판단함
+ 트랜잭션을 지원하는 쓰기 지연(transactional write-behind) : 기본적으로는 트랜잭션을 커밋하는 순간에 INSERT SQL을 DB로 보냄 (영속성 컨텍스트에는 쓰기 지연 SQL 저장소가 존재하며 persist 시 SQL을 생성하여 저장해 둠)
+ 변경 감지(Dirty Checking) : 엔티티 수정 시 따로 데이터베이스에 저장하는 처리가 필요하지 않음 엔티티의 초기 상태를 스냅샷에 저장해둔 후 트랜잭션이 커밋되는 시점에 엔티티와 스냅샷을 비교하여 변경된 경우 UPDATE 쿼리를 생성함
+ 지연 로딩(Lazy Loading)

<br>

### 플러시

플러시 : 영속성 컨텍스트의 변경 내용을 데이터베이스에 반영하는 것 트랜잭션이 커밋될 때 플러시가 일어남 플러시가 진행되어도 영속성 컨텍스트를 비우지 않음

<br>

플러시가 발생하면 일어나는 일

+ 변경 감지
+ 수정된 엔티티를 쓰기 지연 SQL 저장소에 등록함
+ 쓰기 지연 SQL 저장소의 쿼리를 데이터베이스에 전송

<br>

영속성 컨텍스트를 플러시하는 방법

+ 직접 호출 : ```em.flush()```
+ 플러시 자동 호출
  + 트랜잭션 커밋
  + JPQL 쿼리 실행 : 객체를 저장 후 JPQL을 사용하여 조회했을 때 데이터가 조회되지 않는 현상을 방지하기 위해 JPA는 JPQL 실행 시 플러시가 되는 것을 기본 설정으로 함

<br>

플러시 모드 옵션 (```em.setFlushMode(FlushModeType.COMMIT)```)

+ FlushModeType.AUTO : 커밋이나 쿼리를 실행하는 경우 플러시 함 (기본)
+ FlushModeType.COMMIT : 커밋하는 경우에만 플러시 함
